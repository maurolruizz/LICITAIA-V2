ESTRUTURA REAL DO PROJETO — LICITAIA V2
1. Objetivo deste documento

Este documento registra a estrutura física real do repositório do LICITAIA V2, conforme implementado no ambiente de desenvolvimento.

Sua função é servir como fonte oficial de verdade ("as-built") da organização do projeto, garantindo que todas as fases de implementação e manutenção futura utilizem os caminhos reais do repositório, evitando divergências entre arquitetura conceitual e estrutura física.

Este documento é obrigatório para:

preservar regressão zero

evitar criação de estruturas paralelas

garantir continuidade técnica entre chats

permitir que novos desenvolvedores compreendam rapidamente o projeto

alinhar prompts de implementação com o estado real do código

Este documento não substitui os documentos de arquitetura.

Ele registra como o projeto está fisicamente organizado no repositório.

5. Observação arquitetural importante

O frontend do LICITAIA V2 não utiliza a pasta src como raiz de código.

A estrutura do projeto foi organizada com as pastas principais diretamente na raiz do repositório frontend.

Portanto:

todos os caminhos de implementação devem respeitar essa estrutura

nenhum prompt deve criar src/ sem decisão arquitetural explícita

Essa regra evita duplicação de estrutura e quebra de imports.

7. Papel da pasta modules

A pasta modules é reservada para a implementação do núcleo modular do motor de conformidade do LICITAIA V2.

Ela será responsável por abrigar:

A implementação completa desta estrutura começa na FASE 11 do plano de reconstrução do sistema.

8. Regra estrutural oficial para o núcleo modular

A implementação do núcleo modular do LICITAIA V2 deve ocorrer dentro da pasta já existente modules, localizada em:

02-frontend/licitaia-v2-web/modules

Portanto:

não deve ser criada uma pasta src/modules

não deve ser criada estrutura modular paralela

Toda a infraestrutura modular deve ser implementada dentro da pasta existente.

22. Atualização oficial de topologia Git (governança global)

No ciclo de regularização de governança Git, foi consolidada a topologia oficial:

monorepo único na raiz `C:\LICITAIA-V2`.

Situação anterior regularizada:

- havia repositórios aninhados em:
  - `02-frontend/licitaia-v2-demo`
  - `02-frontend/licitaia-v2-web`
  - `03-backend-api/licitaia-v2-api`
  - `04-backend-ai/licitaia-v2-ai-core`
- estes caminhos estavam sob risco de ambiguidade de rastreabilidade no Git da raiz.

Situação oficial vigente:

- o conteúdo desses caminhos foi absorvido para rastreio no Git da raiz;
- não há submódulo formal ativo como topologia oficial;
- não é permitido estado híbrido (gitlink sem submódulo formal) sem norma explícita.

Referência normativa:

- `01-planejamento/governanca/TOPOLOGIA-GIT-OFICIAL-E-REGULARIZACAO.md`

23. Classificação oficial dos frontends para a ETAPA E

Fica estabelecido, para fins de execução da ETAPA E (Frente 6), o seguinte enquadramento canônico:

- `02-frontend/licitaia-v2-web` é o núcleo modular frontend, fonte de contratos, módulos, validações estruturais, eventos e lógica central de conformidade.
- `02-frontend/licitaia-v2-demo` é a camada operacional de interação guiada do usuário e frontend oficial de condução da ETAPA E.

Regra de fronteira obrigatória:

- a camada operacional em `licitaia-v2-demo` é subordinada ao núcleo;
- é proibido duplicar regra normativa central na interface;
- em conflito entre interface e motor, prevalece o motor.

24. Classificação de artefatos estruturais complementares

Ficam classificados os seguintes caminhos físicos:

- `02-frontend/licitaia-v2-web/modules-dist`:
  - artefato derivado técnico de build/distribuição do núcleo modular;
  - não é fonte de verdade arquitetural ou normativa;
  - alterações funcionais devem ocorrer na fonte (`modules`) e não no derivado.

- `.git-backup-raiz`:
  - artefato técnico/administrativo de backup associado à regularização de topologia Git;
  - não participa do runtime do sistema;
  - não constitui camada funcional do produto.

25. Regra de prevenção de bifurcação de frontend

Risco reconhecido:

- a coexistência de dois frontends sem papéis formais gera risco de bifurcação operacional e dupla verdade de comportamento.

Mitigação normativa obrigatória:

- manter `licitaia-v2-demo` como frontend oficial da ETAPA E;
- manter `licitaia-v2-web` como núcleo modular;
- qualquer mudança de papel entre esses caminhos exige atualização prévia dos artefatos:
  - `01-planejamento/ESTRUTURA REAL DO PROJETO — LICITAIA V2.md.txt`
  - `01-planejamento/PLANO-MESTRE-DECYON-V2.md`
  - `01-planejamento/governanca/MATRIZ-DE-FECHAMENTO-DECYON-V2.md`
